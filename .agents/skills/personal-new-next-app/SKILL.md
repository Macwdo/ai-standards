# Personal New Next App

Create a new Next.js app with the default stack and wiring I use in most projects.

Use this skill when the user wants a fresh Next.js codebase and does not want to decide the frontend stack from scratch.

## Defaults

- Package manager: `pnpm`
- Project bootstrap: `pnpm dlx shadcn@latest init`
- Framework: Next.js
- UI foundation: shadcn/ui
- HTTP client: `axios`
- Forms: `react-hook-form` + `zod` + `@hookform/resolvers`
- Server-state: `@tanstack/react-query`
- Client-state: `zustand`

## Hard Rules

- Never add dependencies by editing `package.json` manually.
- Always install runtime packages with `pnpm add ...`.
- Always install dev packages with `pnpm add -D ...`.
- Always run the shadcn CLI with `pnpm dlx shadcn@latest ...`.
- For UI work after bootstrap, use the existing `$shadcn` skill instead of recreating shadcn rules here.
- For React and Next implementation details after bootstrap, also apply `$vercel-react-best-practices`.

## Workflow

1. Derive the app name from the user request.
2. Create the project with shadcn as the bootstrap path:

```bash
pnpm dlx shadcn@latest init --name <app-name> --template next --defaults
```

3. Enter the new project directory.
4. Install the default stack with `pnpm add`:

```bash
pnpm add axios react-hook-form zod @hookform/resolvers @tanstack/react-query zustand
```

5. Verify the app still uses `pnpm` commands for all follow-up installs.
6. Add the minimum starter wiring described below.

## Starter Wiring

After the packages are installed, add the following baseline structure.

### React Query

- Create a shared `QueryClient` provider component for client-side usage.
- Mount it once near the app root.
- Use a small default config. Prefer `retry: 1` and a non-zero `staleTime`.
- Use `axios` inside query functions by default. Do not mix in `fetch` unless the user asks for it.

### Axios

- Create a shared axios instance in a small `lib` module such as `src/lib/axios.ts`.
- Centralize base configuration there so React Query hooks import the same client.
- Keep the instance simple by default. Do not invent interceptors, auth headers, or environment handling unless the task requires them.

### Zustand

- Create one small example store in `src/stores/`.
- Use Zustand for local client state only.
- Do not use Zustand as a replacement for server data that belongs in React Query.

### Forms

- Create one example client form using `react-hook-form` and `zodResolver`.
- Define the schema with `zod` first, then infer the form type from that schema.
- Use shadcn form primitives and inputs when the form UI is part of the requested starter output.

## Next.js Defaults

- Prefer App Router and TypeScript defaults from the shadcn bootstrap.
- Keep components server-first and add `"use client"` only where required for forms, Zustand consumers, and React Query providers.
- Use the project alias already generated by the template. Do not rewrite imports to relative paths without a reason.

## What To Avoid

- Do not manually rewrite `package.json` dependencies.
- Do not add extra libraries such as auth, persistence, state machines, or testing tools unless the user asks.
- Do not add heavy abstraction around axios, React Query, or Zustand during bootstrap.
- Do not create multiple competing API clients or multiple query providers.

## Expected Outcome

The default result should be a working Next.js app created through shadcn, using `pnpm`, with:

- shadcn/ui initialized
- `axios` installed and used as the default HTTP client
- React Query provider wired at the app root
- one small Zustand store
- one example validated form using React Hook Form and Zod

If the user asks for a plain React app instead of Next.js, do not force this skill. Either confirm the change or use a React-specific bootstrap flow.
